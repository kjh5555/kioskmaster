# PRD — 「자판기 어렵지않아요」

> 원래 작업명: **키오스크 연습 미니앱**
> **정식명: 「자판기 어렵지않아요」** — 노인 사용자가 키오스크를 일상적으로 부르는 단어("자판기")를 사용해 친숙성을 높이고, 부정문 격려("어렵지 않아요")로 심리적 진입장벽을 낮춤. 저장소/패키지명 후보: `japangi` (또는 `kiosk-practice`).
>
> **결정 이력 (2026-05-21):** 앱명 확정 · 결제 시뮬레이션 진행 · TTS는 MVP 제외.

---

## 1. 개요

> **한 줄 요약**: 진짜 매장 키오스크와 똑같이 생긴 화면을, 줄 서 있는 사람 눈치 보지 않고 마음껏 눌러볼 수 있는 노인용 연습 미니앱.

대한민국의 패스트푸드점, 카페, 병원, 기차역에서 사람 직원이 키오스크로 빠르게 대체되면서 60대 이상 노인 사용자들이 일상적으로 디지털 소외감을 경험하고 있다. 본 앱은 **Apps in Toss 미니앱(토스 인앱 WebView)** 환경에서 동작하며, 사용자가 실제 매장에서 만날 4가지 대표 키오스크(맥도널드형, 스타벅스형, 병원 무인 접수, KTX 발권)를 **실수해도 괜찮은 연습 환경**으로 시뮬레이션한다. 결제는 시뮬레이션이며 실제 결제는 절대 발생하지 않는다. 목표는 "실제 매장에서 당황하지 않고 끝까지 주문을 마칠 수 있다"는 자신감을 사용자에게 만들어주는 것이다. 토스의 큰 사용자층(특히 토스를 이미 쓰는 노인 자녀 세대 + 토스를 통해 송금하는 노인 본인)에게 자연스럽게 도달할 수 있는 채널을 활용한다.

## 2. 문제 정의 및 동기

### 2.1 사회적 배경
- 패스트푸드, 카페, 영화관, 병원, 공공기관, 기차역까지 **무인 키오스크 도입률**이 가파르게 증가.
- 노인 사용자는 (1) 메뉴 구조가 낯설고, (2) 옵션 화면에서 선택 강제처럼 느껴지고, (3) 뒤에 줄 선 사람 시선이 부담되어 **포기**하거나 **남에게 부탁**하는 일이 빈번.
- 결과: 외출 빈도 감소, 자존감 저하, 디지털 소외감 심화.

### 2.2 기존 대안의 한계
| 대안 | 한계 |
|---|---|
| 가족이 옆에서 가르쳐주기 | 매번 동행 불가, 노인 본인이 부탁을 꺼림 |
| 유튜브 튜토리얼 | 일방향 시청, 직접 손가락으로 눌러보는 경험 부재 |
| 지자체 디지털 교육 | 회차·장소 제약, 매장별 차이 반영 어려움 |
| 실제 매장에서 시도 | 줄 선 사람 압박, 실수 시 결제 발생 위험 |

### 2.3 우리의 차별점
- **WebView 미니앱**으로 설치 진입장벽이 낮음 (이미 깔린 토스 앱 안에서 바로 실행).
- **실제 매장 화면과 시각적으로 유사**한 시뮬레이션 — 익숙해진 동작이 실제 매장으로 그대로 이전.
- **연습 중 배지**, **"실수해도 괜찮아요" 메시지 톤**, **선택형 도움말("?" 버튼)**로 심리적 안전감 확보.

## 3. 대상 사용자 페르소나

### 페르소나 1 — 김순자 (72세, 여성, 서울 거주, 손녀와 외출 자주)
- 스마트폰: 카카오톡, 사진, 토스로 송금 정도. 앱 설치는 자녀가 해줌.
- 상황: 손녀와 맥도널드 갔는데 키오스크 앞에서 한참 머뭇거리다 손녀가 대신 주문. 다음엔 혼자 해보고 싶음.
- 니즈: "내 속도대로, 누가 안 보는 데서, 미리 한 번 해보고 싶다."

### 페르소나 2 — 박영호 (68세, 남성, 부산 거주, 매주 병원 통원)
- 스마트폰: 뉴스, 유튜브, 은행 앱. 글씨 크게 설정.
- 상황: 대학병원 무인 접수기 앞에서 주민번호 키패드 누르다 잘못 눌러서 처음부터 다시. 뒤에 사람 많아 식은땀.
- 니즈: "주민번호 잘못 눌렀을 때 어떻게 지우는지 미리 알고 싶다."

### 페르소나 3 — 이옥분 (78세, 여성, 농촌 거주, 명절마다 KTX 이용)
- 스마트폰: 통화·문자 위주. 토스는 자녀가 용돈 보내줄 때 확인용.
- 상황: 명절에 자녀 집 갈 때 매번 역무원 창구 찾아다님. 키오스크는 "복잡해 보여서" 시도조차 안 함.
- 니즈: "글씨가 크고, 한 번에 너무 많이 보여주지 않으면 좋겠다."

## 4. 목표 및 비목표

### 4.1 Goals (이번 MVP에서 반드시 달성)
- G1. **4개 시나리오** 각각을 처음부터 끝까지 시뮬레이션으로 완주할 수 있다.
- G2. 노인 사용자가 **외부 도움 없이** 홈 → 시나리오 선택 → 완료까지 도달할 수 있다.
- G3. 모든 화면이 **노인 접근성 기준**(큰 글씨, 큰 터치영역, 명확한 대비, 돌아가기 버튼)을 충족한다.
- G4. **"연습 중" 배지**가 모든 시나리오 화면에 항상 보인다 — 실수해도 안전하다는 인식 확보.
- G5. **선택형 도움말("?" 버튼)**이 각 단계마다 제공되어, 막혔을 때 친절한 설명을 볼 수 있다.
- G6. Apps in Toss WebView 환경에서 정상 동작하고, TDS Mobile 디자인 시스템을 일관되게 사용.

### 4.2 Non-Goals (이번 MVP에서 명확히 제외)
- N1. **실제 결제 연동 금지** — 토스 인앱결제 SDK는 호출하지 않는다.
- N2. 4개 외 시나리오(영화관, 주차, 공항 셀프체크인, 무인 편의점 등) — **명시적으로 제외**.
- N3. 다국어(영어, 중국어 등) — **한국어 only**.
- N4. 사용자 진척도 영구 저장(서버 계정 동기화) — 로컬(localStorage)에만 보관. 서버 백엔드 없음.
- N5. 노인 외 사용자(어린이용, 외국인용) 별도 모드.
- N6. 게이미피케이션(뱃지, 점수, 리더보드) — 자신감 향상에 직접 도움 되는지 검증 안 됨, 제외.
- N7. **음성으로 주문하기(STT) 및 음성 안내(TTS)** — 시각적 따라하기와 텍스트 안내에 집중. TTS는 v2 이후 사용자 피드백 따라 재검토.
- N8. 실제 매장 위치 검색·길찾기 — 본질에서 벗어남.

## 5. 사용자 시나리오 (4개)

> 각 시나리오는 **화면 단계 → 핵심 UI 요소 (TDS 매핑) → 노인 사용자 헷갈림 포인트 & 도움 전략 → 성공 기준** 순으로 정리.

---

### 5.1 시나리오 ① 패스트푸드 (맥도널드 타입)

#### 화면 단계
1. 시작 화면 — "주문하시려면 화면을 눌러주세요" (실제 매장 안내 그대로 재현)
2. 매장/포장 선택 — "여기서 드세요" / "포장이에요"
3. 메뉴 카테고리 — 버거 / 세트 / 사이드 / 음료 / 디저트
4. 카테고리 안 메뉴 그리드 — 사진 + 이름 + 가격
5. 단품/세트 선택 — "이 메뉴는 세트로도 드실 수 있어요"
6. 세트 옵션 — 사이즈 (보통/큰사이즈), 사이드 (감자튀김/너겟), 음료 선택
7. 추가 토핑/제외 옵션 (선택 사항 — 건너뛸 수 있음)
8. 장바구니 확인 — 수량 변경, 삭제, 추가 주문 여부
9. 결제 방식 선택 — 카드 / 모바일페이 (시뮬레이션)
10. 결제 시뮬레이션 화면 — 가짜 결제 화면, "결제는 연습이라 실제로 되지 않아요" 안내 강조
11. 주문번호 출력 화면 — "주문번호 A-138 — 받는 곳에서 번호를 불러드릴 거예요"

#### 핵심 UI 요소 (TDS 매핑)
| 단계 | TDS 컴포넌트 |
|---|---|
| 매장/포장 | 큰 `Button` 2개 세로 배치 |
| 카테고리 | `Tab` 또는 큰 카드형 `ListRow` |
| 메뉴 그리드 | 자체 컴포넌트(`MenuCard`) + 큰 사진 + TDS `Badge`(품절/추천) |
| 옵션 화면 | `Checkbox`, `Slider`(사이즈), `BottomSheet`로 옵션 묶음 표시 |
| 장바구니 | `ListRow` + 수량 stepper |
| 결제 방식 | `BottomCTA` + `Modal` 확인 |
| 결제 시뮬레이션 | `Dialog`로 "연습 결제" 안내 |
| 완료 | 큰 타이포 주문번호 화면 |

#### 노인 사용자 헷갈림 포인트 & 도움 전략
- **옵션 강제처럼 보임** → 옵션 화면 상단에 "**그냥 기본으로도 주문할 수 있어요. '다음'을 눌러주세요**" 안내문 고정.
- **세트/단품 차이 모름** → 가격 차이를 큰 글씨로, "세트로 하면 감자튀김+음료가 같이 나와요" 같은 일상어 설명.
- **장바구니 개념 낯섦** → "지금까지 고르신 음식이에요" 헤더 + 수량 +/- 버튼 둘 다 56dp 이상.
- **결제 화면 공포** → 결제 직전 큰 `Dialog`로 "**여기는 연습이에요. 진짜 돈이 나가지 않아요.**" 한 번 더 안내.
- **주문번호 의미 모름** → 마지막 화면에 큰 타이포 + "**주문번호 138번. 매장에서 이 번호가 불리면 음식을 받으세요**" 안내문 + 일러스트.

#### 성공 기준
- ✅ 시작 화면부터 주문번호 출력까지 도달 (이탈 없음)
- ✅ 최소 1회 옵션 선택을 거침 (혹은 "기본으로" 안내 따라 건너뛰기 성공)
- ✅ "연습이에요" 안내가 결제 화면에서 확인됨

### 5.2 시나리오 ② 카페 (스타벅스 타입)

#### 화면 단계
1. 시작 화면 — "주문하시려면 눌러주세요"
2. 매장/포장 선택 — "여기서 마실래요" / "가지고 갈래요"
3. 음료 카테고리 — 커피 / 차 / 음료 / 디저트
4. 음료 선택 — 사진 + 이름 + 가격
5. 사이즈 선택 — Tall / Grande / Venti (한글 음차+설명: "**그란데(보통보다 조금 큰 사이즈)**")
6. 온도 선택 — Hot / Iced
7. 옵션 (얼음양·시럽·샷 등) — 선택 사항, 기본값 명시
8. 장바구니 확인
9. 결제 방식 선택 (시뮬레이션)
10. 결제 시뮬레이션
11. 진동벨/주문번호 화면 — "**진동벨이 울리면 받는 곳으로 가세요**"

#### 핵심 UI 요소 (TDS 매핑)
| 단계 | TDS 컴포넌트 |
|---|---|
| 사이즈 선택 | `Tab` 또는 큰 `Button` 그룹 + 한글 음차 설명 |
| 온도 | 큰 `Button` 2개 (Hot=빨간 톤, Iced=파란 톤) |
| 옵션 | `BottomSheet`에 묶어서, "**기본 그대로 받으실 수 있어요**" 헤더 |
| 진동벨 안내 | 자체 일러스트 + 큰 타이포 주문번호 |

#### 노인 사용자 헷갈림 포인트 & 도움 전략
- **외래어 사이즈명 (Tall/Grande/Venti)** → 모든 사이즈에 한글 음차 + 부피 비유 ("종이컵보다 조금 큰 정도").
- **샷 추가 / 시럽 변경의 의미 모름** → "**원래 들어가는 양이에요. 그대로 두셔도 돼요**" 안내.
- **진동벨 개념 모름** → 일러스트 + "**손에 쥐고 계시다가 울리면 받는 곳으로 가세요**".

#### 성공 기준
- ✅ 음료 1잔을 끝까지 주문해 진동벨 화면 도달
- ✅ 사이즈/온도 선택을 본인이 의식적으로 함

---

### 5.3 시나리오 ③ 병원 무인 접수

#### 화면 단계
1. 시작 화면 — "접수하시려면 눌러주세요"
2. 진료 종류 선택 — "**예약하신 분**" / "**오늘 처음 오신 분 (당일접수)**"
3. 주민등록번호 입력 — 키패드 13자리 (앞 6자리 + 뒷 7자리)
4. 본인 정보 확인 — "○○○님이 맞으세요?" (시뮬레이션 가상 인물)
5. 진료과 선택 — 내과 / 정형외과 / 안과 / ...
6. 의사 선택 — "○○○ 선생님 (오늘 진료 가능)"
7. 접수 확인 — 요약 화면
8. 대기번호 출력 — "**대기번호 12번. 진료실 앞에서 기다려주세요**"

#### 핵심 UI 요소 (TDS 매핑)
| 단계 | TDS 컴포넌트 |
|---|---|
| 진료 종류 | 큰 `Button` 2개 + 보조 설명 |
| 주민번호 입력 | TDS `Keypad` + `TextField` (보안 마스킹: 뒷자리는 ●로) |
| 본인 정보 확인 | `Modal` 또는 큰 카드 + "맞아요" / "다시 입력할게요" |
| 진료과 선택 | `ListRow` 세로 배치 (큰 글씨, 아이콘) |
| 의사 선택 | `ListRow` + 사진 + 진료 시간 |
| 대기번호 | 큰 타이포 + 강조 색상 |

#### 노인 사용자 헷갈림 포인트 & 도움 전략
- **주민번호 잘못 눌렀을 때** → 모든 화면에 큰 "**한 글자 지우기**" 버튼 + "**처음부터 다시**" 버튼.
- **앞자리/뒷자리 구분 어려움** → 화면을 두 줄로 나누고, 입력 위치를 큰 표시로 가이드.
- **개인정보 두려움** → 화면 상단에 "**연습이에요. 진짜로 저장되지 않아요**" 고정 배지.
- **예약/당일접수 차이 모름** → 두 버튼 아래에 한 줄 설명 ("예약하신 분 = 어제까지 전화나 앱으로 미리 신청하신 분").
- **진료과 명칭 어려움** → "내과 (배·소화)", "정형외과 (뼈·관절)" 같이 한글 보조 설명.

#### 성공 기준
- ✅ 주민번호 13자리 입력 완료 (실수 정정 1회 이상 경험 권장)
- ✅ 대기번호 화면 도달

---

### 5.4 시나리오 ④ 지하철/기차 표 (KTX 타입)

#### 화면 단계
1. 시작 화면 — "표를 사시려면 눌러주세요"
2. 출발지 선택 — 자주 가는 역 (서울/부산/대전 등) 큰 버튼 + "다른 역 찾기"
3. 도착지 선택 — 동일 방식
4. 날짜 선택 — 오늘/내일/모레 큰 버튼 + 달력
5. 시간 선택 — 가능한 시간대 목록
6. 인원수 선택 — 어른/어린이 (+/- 스테퍼)
7. 좌석 선택 — 객실 도면 + 좌석 클릭 (또는 "**자리는 알아서 정해주세요**" 버튼)
8. 결제 방식 선택 (시뮬레이션)
9. 결제 시뮬레이션
10. 표 출력 화면 — 출발지·도착지·시간·좌석번호 큰 글씨

#### 핵심 UI 요소 (TDS 매핑)
| 단계 | TDS 컴포넌트 |
|---|---|
| 출발/도착 | 큰 `Button` 그리드 + "다른 역" `BottomSheet` |
| 날짜 | "오늘/내일/모레" 큰 `Button` + 옵션 캘린더 |
| 시간 | `ListRow` 세로 |
| 인원 | +/- `Button` 큰 사이즈 |
| 좌석 | 자체 좌석맵 컴포넌트 + "**알아서 정해주세요**" 옵션 강조 |
| 표 출력 | 표 모양 카드 + 큰 글씨 |

#### 노인 사용자 헷갈림 포인트 & 도움 전략
- **좌석 도면이 어려움** → "**자리는 알아서 정해주세요**" 큰 버튼을 좌석맵 위쪽에 항상 노출.
- **시간대 너무 많음** → 한 화면에 4~5개씩만, 위/아래 스크롤로.
- **어른/어린이 구분** → "어른(만 13세 이상)" 명시.
- **표 출력 후 분실 걱정** → "**역무원에게 보여주거나, 개찰구에 대시면 돼요**" 안내.

#### 성공 기준
- ✅ 출발-도착-날짜-시간-인원-좌석을 모두 선택해 표 출력 화면 도달
- ✅ 좌석 선택 시 "알아서 정해주세요" 옵션을 인지

## 7. 노인 접근성 요구사항

### 7.1 시각
- **기본 글자 크기**: TDS 기본보다 한 단계 큼 — 본문 18~20sp, 버튼 라벨 22~24sp, 화면 헤더 28sp 이상. **[확인 필요]** TDS Mobile이 글자 크기 단계 토큰을 제공하는지 / 커스터마이즈 허용 범위.
- **사용자 추가 확대**: 홈/시나리오 상단에 "글씨 더 크게/조금 작게" 토글 (3단계: 보통/큼/매우 큼). 설정은 localStorage 저장.
- **시각적 대비**: WCAG 2.1 AAA 기준 권장 (본문 텍스트 대비비 7:1 이상).
- **색만으로 정보 전달 금지**: 빨강/초록 외에 아이콘·텍스트 라벨 병행.

### 7.2 터치
- **최소 터치 영역**: 56dp × 56dp, **권장**: 64dp × 64dp 이상.
- **버튼 간 간격**: 최소 12dp.
- **드래그·롱프레스 사용 금지**, 모든 동작은 탭으로 완결.
- **더블탭 사용 금지**, 의도치 않은 탭 방지를 위해 결제·삭제·뒤로가기 등 파괴적 행동은 한 번의 큰 버튼 + 확인 `Dialog`.

### 7.3 청각 (MVP 제외)
- **MVP 제외 결정 (2026-05-21)**: 음성 안내(TTS)는 본 MVP에서 제외한다. 시각 안내·텍스트 톤·도움말 오버레이로 충분히 안내 가능하다고 판단. v2 이후 베타 피드백 따라 재검토.
- 따라서 본 MVP에서는 TTS 관련 SDK·`speechSynthesis`·녹음 mp3 자산 모두 도입하지 않는다.

### 7.4 인지·언어
- **돌아가기/이전 단계** 버튼 항상 명확하게 (좌상단 고정 위치, 큰 화살표 + "이전" 라벨).
- **에러·실수 메시지 톤**: "잘못 누르셨나요? 괜찮아요. 다시 해볼게요" — 비난 없음, 다음 행동 안내 명확.
- **한 화면 정보량 제한**: 동시에 보이는 선택지 최대 5~6개 권장. 더 많으면 스크롤·카테고리 분할.
- **외래어 최소화**: 부득이한 경우 한글 음차 + 짧은 일상어 설명 ("**그란데(보통보다 조금 큰 사이즈)**").
- **연습 모드 강조**: 모든 시나리오 화면 우상단에 "**연습 중**" 배지 고정. 결제 직전에는 큰 `Dialog`로 재확인.
- **선택형 도움말 오버레이**: 매 화면 우상단 "?" 버튼 → 친절한 평어체 설명을 `BottomSheet`로 표시.

### 7.5 모션·애니메이션
- 화면 전환 애니메이션은 짧고 단순하게 (200ms 이하).
- 깜빡이는 요소 금지, 회전 애니메이션 최소화.
- `prefers-reduced-motion` 운영체제 설정 존중 — **[확인 필요]** WebView 환경에서 해당 미디어쿼리가 동작하는지.

### 8.4 아키텍처 원칙
- **시나리오는 독립 모듈**: 각 시나리오 폴더가 자체 단계 라우팅·로컬 상태를 가짐. 한 시나리오 추가/제거가 다른 시나리오에 영향 없도록.

- **데이터는 정적**: 메뉴·역·진료과 등은 TS 파일에 하드코딩. 서버 호출 없음 → 오프라인에서도 동작.

| SDK | 사용 여부 | 이유 |
|---|---|---|
| **인앱결제** | **사용 안 함** | 연습 앱이므로 절대 호출 금지. 결제 화면은 시뮬레이션 UI만. |

| **사용자 식별 키** | **선택, [확인 필요]** | 익명 분석용으로 사용 가능성 검토. 개인정보 수집 최소화 원칙상 보류 권장. |
| **토스 인증** | **사용 안 함** | 본인인증 필요 없음. |
| **토스 광고 픽셀** | **사용 안 함** | 추적 비목표. |
| **WebView 브릿지 기본 API** | **사용** | 뒤로가기 처리, 상태바 색상, 안전영역(safe-area) 처리 등 필수 항목만. |
| **TTS API** | **사용 안 함** | MVP 제외 결정 (2026-05-21). v2에서 재검토. |

> **원칙**: SDK는 "꼭 필요한 것만, 가장 적게" 사용. 노인 사용자에게 추가 권한 요청 화면을 최소화.

## 11. 비기능 요구사항

### 11.1 성능
- **초기 로드(TTI)**: 토스 앱 안 WebView 진입 후 **2초 이내**에 홈 화면 인터랙션 가능. **[확인 필요]** 토스 WebView 콜드 스타트 평균 시간.
- **시나리오 단계 전환**: **300ms 이내** 응답.
- **번들 사이즈**: gzip 후 **200KB 미만** 목표 (이미지·일러스트 제외). 이미지는 `loading="lazy"`.

### 11.2 접근성
- 7장 모든 항목 충족.
- 색상 대비 WCAG AAA.
- 키보드 포커스 핸들링 (WebView에서 외부 키보드 사용 거의 없지만, 시맨틱 마크업은 유지).

### 11.3 호환성
- iOS 토스 인앱 / Android 토스 인앱 모두 동일 동작. **[확인 필요]** Apps in Toss 지원 최소 OS 버전.
- 가로 모드는 지원하지 않음 (세로 고정). granite.config에서 설정 가능 여부 확인.

### 11.4 언어·로컬화
- 한국어 only. 외래어는 한글 음차 + 설명.

### 11.5 개인정보·보안

4. 시나리오 ③ 병원 (가장 정보량 많은 시나리오) 완주
5. 글씨 크기 변경 시도 — 설정 발견 시간 측정
6. 자유 사용 — 나머지 두 시나리오 중 본인 선택

- 시나리오별 평균 완료 시간

## 16. 개발 원칙 (Karpathy 4원칙)

> 본 프로젝트의 모든 개발 작업은 아래 4원칙을 따른다. 코드 리뷰·PR·작업 분할 시 체크리스트로 사용.

### 16.1 Think Before Coding
- 모든 기능 구현 전, **가정을 명시**한다. 모호한 부분은 PRD 작성자/사용자에게 확인 후 진행.
- "왜 이렇게 만드는가"가 PRD·이슈·커밋 메시지 중 하나에 반드시 기록.
- 새 라이브러리·SDK 도입 전 공식 문서를 먼저 읽는다. 추측 금지.

### 16.2 Simplicity First
- **요청한 기능 외 추가 금지**. 좋아 보여서 끼워 넣지 않는다.
- 추측성 추상화 금지 — "나중에 쓸 수도 있어서"라는 이유로 인터페이스·제네릭 도입 금지.
- 외부 상태 라이브러리, 복잡한 데이터 페칭 도구, 큰 UI 의존성 — MVP에는 도입하지 않는다.

### 16.3 Surgical Changes
- 작업 중인 함수·컴포넌트 외 **인접 코드 임의 수정 금지**. 리팩토링이 필요하면 별도 PR.
- 기존 코딩 스타일·네이밍·들여쓰기를 그대로 따른다.
- 한 PR은 **하나의 의도**만 담는다.

### 16.4 Goal-Driven Execution
- 모든 작업은 **검증 가능한 성공 기준**으로 변환된다 (예: "패스트푸드 시나리오의 5단계 옵션 화면이 보이고, 옵션 안 고르고도 다음으로 갈 수 있다").
- "구현했다"는 빌드 성공이 아니라 **검증 완료** 시점이다.
- 완료 주장 전에 항상 신선한 증거(빌드 통과, 테스트 통과, 베타 사용자 확인)를 제시한다.
